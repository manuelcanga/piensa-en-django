Sección 01

01.¿Qué es una relación One-to-One?

Una relación One-to-One (uno a uno) es el tipo más estricto de relación entre tablas. Significa que un registro de una tabla está asociado con exactamente un registro de otra tabla, y viceversa. Ni más, ni menos: uno y solo uno.

RelaciónDescripciónEjemploCampo en Django
One-to-One Un registro ↔ exactamente uno Usuario ↔ Perfil OneToOneField
One-to-Many Un registro → muchos del otro Autor → Libros ForeignKey
Many-to-Many Muchos ↔ muchos Estudiantes ↔ Cursos ManyToManyField

Piensa en OneToOneField como un ForeignKey con una restricción extra: la combinación de valores en la columna FK debe ser única. Cada padre tiene a lo sumo un hijo, y cada hijo tiene exactamente un padre.

→ Internamente Django añade unique=True a la columna FK en la base de datos
💡

El uso más clásico de OneToOneField en Django es extender el modelo de usuario incorporado (auth.User) con campos adicionales mediante un modelo Perfil. Es el patrón recomendado por la documentación oficial de Django.

Sección 02

02.One-to-One vs ForeignKey: ¿en qué se diferencian?

Como acabamos de ver, OneToOneField se parece mucho a ForeignKey. La diferencia clave está en la unicidad:

⬡ OneToOneField

La columna FK tiene una restricción UNIQUE en la base de datos. Cada registro padre puede tener como máximo un registro hijo. Django lanza un error si intentas crear un segundo hijo para el mismo padre.

⬡ ForeignKey

La columna FK no es única. Cada registro padre puede tener muchos hijos. Es la relación One-to-Many habitual.

Diferencia conceptual Python 
# ── ForeignKey: un autor puede tener MUCHOS libros ──
class Libro(models.Model):
    autor = models.ForeignKey(Autor, on_delete=models.CASCADE)
    # autor_id no es único → un autor puede aparecer muchas veces


# ── OneToOneField: un usuario tiene UN SOLO perfil ──
class Perfil(models.Model):
    usuario = models.OneToOneField(Usuario, on_delete=models.CASCADE)
    # usuario_id es ÚNICO → cada usuario solo puede tener 1 perfil
    # Si intentas crear un segundo: IntegrityError (UNIQUE constraint failed)
✅

Una forma fácil de elegir: si la pregunta es "¿puede este padre tener más de uno?" y la respuesta es NO, usa OneToOneField. Si la respuesta es SÍ, usa ForeignKey.

Sección 04

04.OneToOneField en Django

Vamos a construir nuestro primer ejemplo: un sistema de usuarios con perfil. Usaremos el User de Django como modelo base y le añadiremos un perfil con datos extra.

Definir el modelo

usuarios/models.py Python · Django 
from django.db import models
from django.contrib.auth.models import User


class Perfil(models.Model):
    # 👇 OneToOneField: la clave de la relación uno a uno
    usuario = models.OneToOneField(
        User,
        on_delete=models.CASCADE,     # si se borra el User, se borra el Perfil
        related_name='perfil',        # acceso: user.perfil
        verbose_name='Usuario'
    )

    # Campos extra que no tiene el User por defecto
    avatar      = models.ImageField(upload_to='avatares/', null=True, blank=True)
    bio         = models.TextField(blank=True)
    telefono    = models.CharField(max_length=20, blank=True)
    ciudad      = models.CharField(max_length=100, blank=True)
    sitio_web   = models.URLField(blank=True)
    fecha_nac   = models.DateField(null=True, blank=True)

    class Meta:
        verbose_name        = 'Perfil'
        verbose_name_plural = 'Perfiles'

    def __str__(self):
        return f"Perfil de {self.usuario.username}"

Parámetros de OneToOneField

ParámetroObligatorioDescripción
to (1er arg)✅ SíModelo al que apunta la relación
on_delete✅ SíQué hacer si se borra el padre
related_name❌ NoNombre del acceso inverso (muy recomendado)
null=True❌ NoLa FK puede ser NULL (relación opcional)
blank=True❌ NoPermite campo vacío en formularios
parent_link❌ NoIndica herencia de modelo (avanzado)
💡

OneToOneField acepta exactamente los mismos parámetros que ForeignKey (excepto que no tiene limit_choices_to de la misma forma y añade parent_link). Todo lo que sabes de FK se aplica aquí.

Sección 05

05.on_delete: comportamiento al borrar el padre

Al igual que con ForeignKey, debes decidir qué ocurre con el objeto hijo cuando se elimina el padre. Las opciones son las mismas:

CASCADE
Borra también el perfil cuando se borra el usuario. El más común en O2O: si el usuario desaparece, su perfil no tiene sentido.
PROTECT
Impide borrar el usuario si existe su perfil. Lanza ProtectedError. Útil si el perfil contiene datos que no quieres perder accidentalmente.
SET_NULL
Pone la FK del perfil a NULL. El perfil queda "huérfano". Requiere null=True en el campo. Caso inusual en O2O.
SET_DEFAULT
Asigna el valor por defecto al campo. Requiere default=....
DO_NOTHING
No hace nada en Django. Puede causar errores de integridad en la BD. Evítalo salvo que gestiones la integridad manualmente.
⚠️

En casi todos los casos de OneToOneField la opción correcta es CASCADE: si el padre no existe, el hijo tampoco tiene sentido. Es raro necesitar otra opción en O2O.

Sección 07

07.Operaciones básicas con el ORM

Crear objetos con OneToOneField

Django Shell / views.py Python 
from django.contrib.auth.models import User
from .models import Perfil

# ── Crear un usuario ──
user = User.objects.create_user(
    username='ana',
    email='ana@ejemplo.com',
    password='clave1234'
)

# ── Crear su perfil (vinculado con O2O) ──
perfil = Perfil.objects.create(
    usuario   = user,
    bio       = "Desarrolladora web apasionada",
    ciudad    = "Madrid",
    telefono  = "600 000 000"
)

# ── Intentar crear un segundo perfil para el mismo usuario ──
# 💥 Lanza IntegrityError: UNIQUE constraint failed
Perfil.objects.create(usuario=user, ciudad="Barcelona")

# ── Acceder al perfil DESDE el usuario (acceso inverso) ──
print(user.perfil.ciudad)      # → "Madrid"
print(user.perfil.bio)         # → "Desarrolladora web apasionada"
print(user.perfil.usuario_id)  # → 1 (el id del user, sin query)

# ── Acceder al usuario DESDE el perfil ──
print(perfil.usuario.username) # → "ana"
print(perfil.usuario.email)    # → "ana@ejemplo.com"

# ── Modificar el perfil ──
perfil.ciudad    = "Sevilla"
perfil.sitio_web = "https://ana.dev"
perfil.save()

# ── Borrar el perfil (no borra el user con CASCADE definido así) ──
perfil.delete()   # solo borra el perfil

# ── Borrar el user (con CASCADE sí borra el perfil también) ──
user.delete()    # borra user Y perfil

El acceso inverso: de padre a hijo

A diferencia de ForeignKey (donde el acceso inverso devuelve un Manager con muchos objetos), en OneToOneField el acceso inverso devuelve directamente el objeto, no un QuerySet.

OneToOneField (O2O)

El acceso inverso devuelve el objeto directamente:
user.perfil → objeto Perfil
No necesitas .all() ni .first()

ForeignKey (FK)

El acceso inverso devuelve un Manager:
autor.posts → Manager
Necesitas .all(), .filter()…

La excepción RelatedObjectDoesNotExist

Si intentas acceder al perfil de un usuario que no tiene perfil, Django lanza una excepción especial. Debes gestionarla:

Gestión de la excepción Python 
# ❌ Si el usuario no tiene perfil, esto lanza excepción
user.perfil   # → RelatedObjectDoesNotExist !

# ✅ Opción 1: try/except
try:
    perfil = user.perfil
except Perfil.DoesNotExist:
    perfil = None

# ✅ Opción 2: hasattr (comprobación previa)
if hasattr(user, 'perfil'):
    print(user.perfil.ciudad)
else:
    print("Este usuario no tiene perfil")

# ✅ Opción 3: get_or_create (crea el perfil si no existe)
perfil, creado = Perfil.objects.get_or_create(usuario=user)
if creado:
    print("Se ha creado un perfil vacío")

# ✅ Opción 4: Perfil.objects.filter (no lanza excepción)
perfil = Perfil.objects.filter(usuario=user).first()  # None si no existe
🚨

¡Cuidado! Este es el error más frecuente con OneToOneField. Si la relación puede no existir (el usuario puede no tener perfil), protege siempre el acceso con try/except, hasattr() o get_or_create().

Sección 08

08.Consultas avanzadas con OneToOneField

El sistema de consultas de Django funciona exactamente igual que con ForeignKey. Usamos el doble guión bajo __ para cruzar la relación en cualquier dirección.

Consultas ORM avanzadas Python 
from django.contrib.auth.models import User
from django.db.models import Q

# ── 1. Usuarios que tienen ciudad = Madrid en su perfil ──
User.objects.filter(perfil__ciudad="Madrid")

# ── 2. Usuarios con bio rellena ──
User.objects.filter(perfil__bio__gt="")

# ── 3. Usuarios SIN perfil (perfil es NULL / no existe) ──
User.objects.filter(perfil__isnull=True)

# ── 4. Usuarios CON perfil ──
User.objects.filter(perfil__isnull=False)

# ── 5. Perfiles de usuarios activos, ordenados por username ──
Perfil.objects.filter(usuario__is_active=True)\
       .order_by('usuario__username')

# ── 6. Perfiles de Madrid O con sitio web ──
Perfil.objects.filter(
    Q(ciudad="Madrid") | Q(sitio_web__gt="")
)

# ── 7. Buscar perfil por email del usuario ──
Perfil.objects.get(usuario__email="ana@ejemplo.com")

# ── 8. Todos los perfiles de usuarios con nombre "Ana" ──
Perfil.objects.filter(usuario__first_name__icontains="Ana")

# ── 9. ¿Existe el perfil de un usuario? ──
Perfil.objects.filter(usuario=user).exists()  # → True / False
Sección 09

09.Optimización: select_related

OneToOneField, como ForeignKey, genera una query adicional cada vez que accedes al objeto relacionado. select_related resuelve esto con un JOIN:

select_related con O2O Python 
# ❌ MAL: N+1 queries
perfiles = Perfil.objects.all()
for p in perfiles:
    print(p.ciudad, p.usuario.username)  # 💥 query por cada p.usuario

# ✅ BIEN: 1 sola query con JOIN
perfiles = Perfil.objects.select_related('usuario')
for p in perfiles:
    print(p.ciudad, p.usuario.username)  # ✅ ya en memoria

# También en sentido inverso: usuarios con su perfil ya cargado
usuarios = User.objects.select_related('perfil')
for u in usuarios:
    # Cuidado: si no tiene perfil, acceder a u.perfil lanza excepción
    try:
        print(u.username, u.perfil.ciudad)
    except Perfil.DoesNotExist:
        print(u.username, "sin perfil")
✅

Usa siempre select_related en las vistas de listado cuando vayas a acceder a la relación O2O dentro de un bucle. La diferencia de rendimiento puede ser enorme con muchos registros.

Sección 10

10.Caso práctico: el patrón Perfil de Usuario

Este es el uso más habitual de OneToOneField en Django. El User de Django tiene campos básicos (username, email, password…), pero casi siempre necesitas más: avatar, bio, teléfono, redes sociales…

La solución elegante es crear un modelo Perfil vinculado con OneToOneField y crear el perfil automáticamente cada vez que se crea un usuario, usando señales (signals).

Creación automática con señales (signals)

usuarios/signals.py Python · Django Signals 
from django.db.models.signals import post_save
from django.dispatch import receiver
from django.contrib.auth.models import User
from .models import Perfil


@receiver(post_save, sender=User)
def crear_perfil_usuario(sender, instance, created, **kwargs):
    """Se ejecuta automáticamente DESPUÉS de guardar un User."""
    if created:
        # Solo al CREAR (no al actualizar) el usuario
        Perfil.objects.create(usuario=instance)


@receiver(post_save, sender=User)
def guardar_perfil_usuario(sender, instance, **kwargs):
    """Guarda el perfil cuando se guarda el usuario."""
    try:
        instance.perfil.save()
    except Perfil.DoesNotExist:
        pass

Ahora debes registrar las señales en el archivo de configuración de la app:

usuarios/apps.py Python 
from django.apps import AppConfig


class UsuariosConfig(AppConfig):
    default_auto_field = 'django.db.models.BigAutoField'
    name = 'usuarios'

    def ready(self):
        # Importar aquí para registrar las señales al arrancar Django
        import usuarios.signals  # noqa: F401
✅

Con las señales configuradas, cada vez que se cree un usuario (desde el Admin, desde una vista, desde el shell…), Django creará automáticamente su perfil vacío. ¡No tienes que acordarte de hacerlo manualmente!

Vista de edición de perfil

Un patrón muy común es mostrar en la misma página el formulario del usuario y el de su perfil:

usuarios/views.py Python · Django Views 
from django.contrib.auth.decorators import login_required
from django.shortcuts import render, redirect
from .forms import UserForm, PerfilForm


@login_required
def editar_perfil(request):
    if request.method == 'POST':
        # Formulario del User (datos básicos)
        user_form   = UserForm(request.POST, instance=request.user)
        # Formulario del Perfil (datos extra)
        perfil_form = PerfilForm(
            request.POST,
            request.FILES,             # para el avatar (ImageField)
            instance=request.user.perfil
        )
        if user_form.is_valid() and perfil_form.is_valid():
            user_form.save()
            perfil_form.save()
            return redirect('mi_perfil')
    else:
        user_form   = UserForm(instance=request.user)
        perfil_form = PerfilForm(instance=request.user.perfil)

    return render(request, 'editar_perfil.html', {
        'user_form':   user_form,
        'perfil_form': perfil_form,
    })

11.Formularios con OneToOneField

usuarios/forms.py Python · Django Forms 
from django import forms
from django.contrib.auth.models import User
from .models import Perfil


class UserForm(forms.ModelForm):
    """Formulario para editar datos básicos del User."""
    class Meta:
        model  = User
        fields = ['first_name', 'last_name', 'email']


class PerfilForm(forms.ModelForm):
    """Formulario para editar el Perfil (datos extra)."""
    class Meta:
        model  = Perfil
        # Excluimos el campo 'usuario' porque ya lo tenemos del request
        exclude = ['usuario']
        widgets = {
            'bio': forms.Textarea(attrs={'rows': 4}),
            'fecha_nac': forms.DateInput(attrs={'type': 'date'}),
        }
Sección 12

12.Django Admin con OneToOneField

El Admin de Django también funciona muy bien con relaciones O2O. Puedes mostrar el perfil directamente dentro del formulario del usuario usando un StackedInline:

usuarios/admin.py Python · Django Admin 
from django.contrib import admin
from django.contrib.auth.admin import UserAdmin
from django.contrib.auth.models import User
from .models import Perfil


class PerfilInline(admin.StackedInline):
    """El perfil aparece integrado en el formulario del User."""
    model         = Perfil
    can_delete    = False   # no permitir borrar el perfil desde aquí
    verbose_name_plural = 'Perfil'
    fields        = ['bio', 'ciudad', 'telefono', 'sitio_web', 'fecha_nac']


class UserConPerfilAdmin(UserAdmin):
    inlines = [PerfilInline]

    def get_inline_instances(self, request, obj=None):
        # No mostrar el inline al crear un nuevo usuario
        if not obj:
            return []
        return super().get_inline_instances(request, obj)


# Reemplazamos el UserAdmin por defecto con el nuestro
admin.site.unregister(User)
admin.site.register(User, UserConPerfilAdmin)


# También registramos Perfil por separado (por si se necesita)
@admin.register(Perfil)
class PerfilAdmin(admin.ModelAdmin):
    list_display        = ['usuario', 'ciudad', 'telefono']
    search_fields       = ['usuario__username', 'ciudad']
    list_select_related = ['usuario']
    raw_id_fields       = ['usuario']
✅

Usar StackedInline (en vez de TabularInline) para un O2O tiene sentido porque solo hay un perfil por usuario. StackedInline muestra los campos en un formato más expandido y cómodo.

Sección 13

13.Errores comunes y cómo evitarlos

!

No proteger el acceso al objeto relacionado

Acceder a user.perfil sin try/except cuando el perfil puede no existir. Siempre protege el acceso o garantiza la existencia del perfil (con señales o get_or_create).

!

Confundir el acceso inverso con un Manager

En O2O, user.perfil devuelve el objeto directamente (no un Manager). No uses user.perfil.all() ni user.perfil.first(). Eso es para ForeignKey.

!

Usar O2O cuando necesitas FK

Si un usuario puede tener varias direcciones, varias fotos, varios pedidos… ¡eso no es O2O! Usa ForeignKey. O2O es solo para la relación absolutamente exclusiva.

!

Olvidar registrar las señales

Si defines señales en signals.py pero no las importas en apps.py dentro del método ready(), nunca se ejecutarán. Es un error muy silencioso y difícil de detectar.

!

No usar select_related en bucles

Como con FK, si accedes al objeto relacionado dentro de un bucle sin select_related, estarás generando una query por cada iteración. El problema N+1 afecta igual a O2O.

Sección 14

14.Buenas prácticas

1

Define siempre related_name con nombre singular

A diferencia de FK (donde usas plural, como 'posts'), en O2O usa singular: related_name='perfil', related_name='expediente'. Es más natural porque el acceso devuelve un único objeto.

2

Usa señales para garantizar la existencia del hijo

Si el objeto hijo siempre debe existir junto al padre, crea el hijo automáticamente con una señal post_save. Así nunca tendrás que comprobar si existe.

3

Protege siempre el acceso inverso

Si no puedes garantizar que el objeto hijo siempre existe, usa try/except, hasattr() o filter().first(). Nunca accedas directamente sin protección.

4

Usa select_related en vistas de listado

En cualquier vista que liste objetos y necesite datos del modelo relacionado, usa siempre select_related para evitar el problema N+1.

5

Coloca el OneToOneField en el modelo "extensión"

El campo siempre va en el modelo secundario (el "hijo"), no en el principal. El Perfil tiene la FK al User, no al revés. Así el modelo principal queda limpio.

6

Considera AbstractUser para casos complejos

Si necesitas añadir muchos campos al usuario o cambiar su comportamiento profundamente, considera extender AbstractUser en vez de usar O2O. Es más potente, pero más complejo y hay que definirlo al inicio del proyecto.
